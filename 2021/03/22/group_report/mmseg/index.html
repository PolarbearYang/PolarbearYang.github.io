语义分割模块(mmseg)

1. 引言

该模块包含的内容主要有：

  • 各种语义分割方法和预训练模型
  • 统一的推断流水线
  • 可以对单图推断产生demo并可视化

每一个任务的核心都是配置文件config.py，所有流程都将依赖这个文件执行。

因此本文档主要介绍的是配置文件的内容，从数据集，到模型，最后是如何使用此配置文件（超参和其他配置部分与设计文档中超参相关配置和其他相关配置相同，因此不再描述）。

该任务大类下只有语义分割一种任务。

如果对如何制定配置文件不感兴趣，只关注如何使用默认配置跑通一个流程，请直接跳至第五节-命令行调用部分。

2. 数据集配置的制定

目前主要使用并且已部署的数据集包括：

  • Cityscapes
  • ADE20K
  • Pascal Context(Pascal VOC 2010)
  • Pascal VOC 2012 + Aug
  • CHASE_DB1
  • HRF(高分辨率指纹)
  • STARE(血管)

每个数据集都有自己的解析类，但这只对配置文件中type属性和标注文件ann_file的选择有影响，大体不影响流水线制定。

2.1 数据集配置

以下是一个Cityscapes数据集进行语义分割的配置例子：

  • samples_per_gpu 每个gpu上分配多少个样本(图片), 该数乘以我们实际使用的gpu数就是batch size
  • workers_per_gpu 每个gpu上分配多少个worker
  • train/val/test 分别对应训练流水线/验证流水线/测试流水线，它还有以下几个重要参数：
    • type 对应用于解析数据集的python类的名称
    • data_root 对应type指明的类，所有数据所在的文件夹
    • img_dir 对应于type指明的类，在相对于data_root指明的路径下，原图像文件的路径
    • ann_dir 对应于type指明的类，在相对于data_root指明的路径下，标注文件的路径(标注文件可以是json或者图像)
    • pipeline 流水线的制定，除去读取和输出数据的部分，主要是一些数据预处理手段

前面的例子为了方便查看及使用，将pipeline信息写在data外面，以预定义的形式展现，这将在之后展开说明。pipeline用一个list来展示。

此外，evaluation模块实际上独立于data模块。但由于和数据集的选择高度相关，从而也写在数据集配置文件中。这里的evaluation只包含interval和metric两个参数：

  • interval表示多少个iter进行一次验证（语义分割任务中一般使用IterBasedRunner）
  • metric表示使用的度量类型（一般使用metric='mIoU'即可）

2.2 流水线配置

流水线用一个list，表示的是对数据依次进行的预处理操作。其中有两个组件是测试/验证/测试阶段均必要的：

  • LoadImageFromFile表示从data_prefix对应的文件夹下载入图片的过程。必须放在流水线的第一步。
  • Collect表示对结果格式的统一。这一步进行后，数据便可以以正确的格式进入模型。必须放在最后一步。

其中训练阶段和验证/测试阶段略有差异：

  • 训练阶段需要添加LoadAnnotations表示对标注文件的读取。一般放在流水线的第二步。
  • 在最后一步collect之前，训练阶段应当使用DefaultFormatBundle两步将图像、语义分割信息均转化为PyTorch的Tesnor格式，验证/测试阶段只需使用ImageToTensor转化图片即可。
  • 验证/测试阶段不应当对图像进行任何随机变换之类的数据增强操作，其他应与训练阶段完全一致。

3. 模型配置的制定

如果要使用自己的数据集，且对模型内部其他细节一概不关心的话，在这一部分要修改的只有两项：decode_head以及auxiliary_head下的num_classes！

语义分割模型的结构大致都相似，其组件内容为：

  • backbone
  • decode_head
  • auxiliary_head(optional)

唯一的例外是PointRend中在backbone的基础上额外加了一个neck(FPN)。

  • decode_head在Backbone提取特征的基础上，将特征解码为语义分割结果。其类型主要由type决定（type即python类名），其他参数都是对应于该模块类的参数。
  • auxiliary_head与decode_head功能相似，将特征解码为语义分割结果，以此辅助decode_head解码。

5. 命令行调用部分

要完成一个任务，主要分为训练、测试、log分析、结果分析四个阶段。一个任务完成之后，还可以根据训练好的模型和自定义素材产生demo。以下分别介绍这四个任务阶段和生成demo调用的接口。

5.1 训练阶段dist_train.sh

该文件位于mmlab/mmsegmentation/tools/dist_train.sh，其参数如下：

以上参数中只有WORK_DIR是必须手动指定的，其他参数都可以由系统自动指定。

以下是一些关于参数的具体的说明：

5.1.1 必要参数

  • WORK_DIR 参数为任务路径。

5.1.2 主要可选参数

  • GPUS 使用的GPU数量，默认值为1。
  • CONFIG 使用的config文件，默认使用工作路径下的config.py文件。
  • PORT （可选）使用的端口编号。格式为PORT:-29500。

5.1.3 次要可选参数

  • --load-from （可选）载入的预训练参数文件路径。
  • --resume-from （可选）如果有之前中断的训练xxx.pth文件，参数为该xxx.pth文件的路径。从该模型继续训练。
  • --no-validate （可选）无参数。若有此选项，则训练时不显示验证集结果。
  • --deterministic （可选）无参数。CUDNN中backend中的某个选项。
  • --options （可选）参数为多个具有形式xxx=yyy的参数。这将被合并到config.py中。
  • --local_rank （可选）参数是一个int。似乎与分布式训练相关。

5.2 测试阶段dist_test.sh

该文件位于mmlab/mmsegmentation/tools/dist_test.sh，其参数如下：

5.2.1 必要参数

  • WORK_DIR 参数为任务路径。
  • EVAL 参数为使用的度量类型，可以使用多个。对Cityscapes数据集可使用cityscapes，对其他数据集可使用mIoU。

5.2.2 主要可选参数

  • GPUS 使用的GPU数量，默认值为1。
  • CONFIG 使用的config文件，默认使用工作路径下的config.py文件。
  • PORT （可选）使用的端口编号。格式为PORT:-29500。

5.2.3 次要可选参数

  • --aug-test 若使用此选项，则测试时也进行数据增强。
  • --out 参数为存储测试结果的*.pkl或*.json文件的路径。
  • --show （暂时无法使用）无参数。显示结果。
  • --show-dir （暂时无法使用）用于显示结果图片存放的目录位置。
  • --gpu-collect 决定是否使用GPU收集结果。
  • --tmpdir 无参数。是否使用临时文件来写出一些结果
  • --options 参数为一至多个xxx=yyy格式改写的config参数。这将被合并到config中。
  • --eval-options 参数为一至多个xxx=yyy格式改写的度量选项。参考dataset.evaluate()。
  • --local_rank 参数是一个int。似乎与分布式训练相关。

6. 几个样例

以下部分显示了一个完整的训练-测试-log分析-结果分析-DEMO测试流程以供用户使用。

目前log分析和结果分析功能暂不可用。

当该流程结束后，预计文件架构如下（以下仅仅是一个例子）：

  • config.py 包含了调用的数据集、模型等信息的配置文件
  • training 包含了训练产生的临时文件、log文件、以及训练好的模型参数的文件夹。
    • log文件以训练开始的时间为名称，存储了每一个epoch/iter的训练及验证信息
    • latest.pth即为每一个epoch/iter结束后训练好的参数文件保存
  • result.pkl 由测试进程产生，包含了验证/测试集图片上的测试信息
  • log_analysis 包含对log进行分析产生的结果。
  • result_analysis 包含测试结果。具体细节待定。
  • demo文件夹下存放用户使用部分数据集之外的图片测试之后产生的结果
