图像分类模块(mmcls)

1. 引言

该模块包含的内容主要有：

  • 单标签分类和多标签分类
  • 各种backbone和预训练模型
  • tricks的集合
  • 可以对单图推断产生demo并可视化

每一个任务的核心都是配置文件config.py，所有流程都将依赖这个文件执行。

因此本文档主要介绍的是配置文件的内容，从数据集，到模型，最后是如何使用此配置文件（超参和其他配置部分与设计文档中超参相关配置和其他相关配置相同，因此不再描述）。

单标签分类和多标签分类使用的模型架构大致相同，其唯一的区别在于其使用的cls_head不同（单标签使用LinearClsHead，多标签使用MultiLabelClsHead）。

如果对如何制定配置文件不感兴趣，只关注如何使用默认配置跑通一个流程，请直接跳至第五节-命令行调用部分。

2. 数据集配置的制定

目前主要使用并且已部署的数据集包括：

  • ImageNet(ILSVRC2012)
  • CIFAR10
  • CIFAR100
  • MNIST
  • FashionMNIST
  • Pascal VOC 2007(Multi-Label)

每个数据集都有自己的解析类，但这只对配置文件中type属性和标注文件ann_file的选择有影响，大体不影响流水线制定。

2.1 数据集配置

和数据集预处理有关的模块主要集中在data模块中，其中至少应包含以下五项：

  • samples_per_gpu 每个gpu上分配多少个样本(图片), 该数乘以我们实际使用的gpu数就是batch size
  • workers_per_gpu 每个gpu上分配多少个worker
  • train/val/test 分别对应训练流水线/验证流水线/测试流水线，它还有以下几个重要参数：
    • type 对应用于解析数据集的python类的名称
    • data_prefix 对应type指明的类，图像数据所在的文件夹
    • ann_file 对应于type指明的类，选择的标注文件的路径
    • pipeline 流水线的制定，除去读取和输出数据的部分，主要是一些数据预处理手段

前面的例子为了方便查看及使用，将pipeline信息写在data外面，以预定义的形式展现，这将在之后展开说明。pipeline用一个list来展示。

此外，evaluation模块实际上独立于data模块。但由于和数据集的选择高度相关，从而也写在数据集配置文件中。这里的evaluation只包含interval和metric两个参数：

  • interval表示多少个epoch进行一次验证（一般只使用EpochBasedRunner）
  • metric表示使用的度量类型（对于单标签分类，metric='accuracy'；对于多标签分类，metric=['mAP', 'CP', 'OP', 'CR', 'OR', 'CF1', 'OF1']）

多标签分类的配置格式除了evaluation项外与单标签分类完全相同，这里不再列出。

2.2 流水线配置

流水线用一个list，表示的是对数据依次进行的预处理操作。其中有两个组件是测试/验证/测试阶段均必要的：

  • LoadImageFromFile表示从data_prefix对应的文件夹下载入图片的过程。必须放在流水线的第一步。
  • Collect表示对结果格式的统一。这一步进行后，数据便可以以正确的格式进入模型。必须放在最后一步。

其中训练阶段和验证/测试阶段略有差异：

  • 在最后一步collect之前，训练阶段应当使用ImageToTensor和ToTesnor两步将图像和标签均转化为PyTorch的Tesnor格式，验证/测试阶段只需转化图片即可
  • 验证/测试阶段不应当对图像进行任何随机变换之类的数据增强操作，其他应与训练阶段完全一致。

3. 模型配置的制定

如果要使用自己的数据集，且对模型内部其他细节一概不关心的话，在这一部分要修改的只有两项：head或backbone下的num_classes！

图像分类的组件内容是所有任务中最简单的，且单标签分类和多标签分类没有差别，组件内容为：

  • 目前可用的Neck只有GlobalAveragePooling，如果不想使用池化则设为None即可。
  • head主要作用是区分单标签和多标签分类（这用LinearClsHead(有Neck)/ClsHead(无Neck)和MultiLabelLinearClsHead(有Neck)/MultiLabelClsHead(无Neck)来区分）。一般情况下修改该组件中的num_classes属性来适应不同的数据集，in_channels应当与backbone中的输出一致。

5. 命令行调用部分

要完成一个任务，主要分为训练、测试、log分析、结果分析四个阶段。一个任务完成之后，还可以根据训练好的模型和自定义素材产生demo。以下分别介绍这四个任务阶段和生成demo调用的接口。

5.1 训练阶段dist_train.sh

该文件位于mmlab/mmclassification/tools/dist_train.sh，其参数如下：

以上参数中只有WORK_DIR是必须手动指定的，其他参数都可以由系统自动指定。

以下是一些关于参数的具体的说明：

5.1.1 必要参数

  • WORK_DIR 参数为任务路径。

5.1.2 主要可选参数

  • GPUS 使用的GPU数量，默认值为1。
  • CONFIG 使用的config文件，默认使用工作路径下的config.py文件。
  • PORT （可选）使用的端口编号。格式为PORT:-29500。

5.1.3 次要可选参数

  • --resume-from 如果有之前中断的训练xxx.pth文件，参数为该xxx.pth文件的路径。从该模型继续训练。
  • --no-validate 无参数。若有此选项，则训练时不显示验证集结果。
  • --seed 参数为使用的随机数种子，int类型。
  • --deterministic 无参数。CUDNN中backend中的某个选项。
  • --options 参数为多个具有形式xxx=yyy的参数。这将被合并到config.py中。
  • --local_rank 参数是一个int。似乎与分布式训练相关。

5.2 测试阶段dist_test.sh

该文件位于mmlab/mmclassification/tools/dist_test.sh，其参数如下：

5.2.1 必要参数

  • WORK_DIR 参数为任务路径。
  • METRIC 参数为使用的度量类型，可以使用多个。对单标签数据集可使用accuracy, precision, recall, f1_score, support，对多标签数据集可使用mAP, CP, CR, CF1, OP, OR, OF1。

5.2.2 主要可选参数

  • GPUS 使用的GPU数量，默认值为1。
  • CONFIG 使用的config文件，默认使用工作路径下的config.py文件。
  • PORT （可选）使用的端口编号。格式为PORT:-29500。

5.2.3 次要可选参数

  • --out 参数为存储测试结果的*.pkl或*.json文件的路径。
  • --show （暂时无法使用）无参数。显示结果。
  • --show-dir （暂时无法使用）用于显示结果图片存放的目录位置。
  • --gpu-collect 决定是否使用GPU收集结果。
  • --tmpdir 是否使用临时文件来写出一些结果
  • --options 参数为一至多个xxx=yyy格式改写的config参数。这将被合并到config中。
  • --metric-options 参数为一至多个xxx=yyy格式改写的度量选项。参考dataset.evaluate()。
  • --show-options （暂时无法使用）参数为一至多个xxx=yyy格式改写的用于显示结果的选项。参考model.show_result。
  • --local_rank 参数是一个int。似乎与分布式训练相关。

5.3.1 必要参数

  • --style 参数为plt作图的风格，默认为'dark'

以下部分显示了一个完整的训练-测试-log分析-结果分析-DEMO测试流程以供用户使用。

目前结果分析功能暂不可用。

当该流程结束后，预计文件架构如下（以下仅仅是一个例子）：

  • config.py 包含了调用的数据集、模型等信息的配置文件
  • training 包含了训练产生的临时文件、log文件、以及训练好的模型参数的文件夹。
    • log文件以训练开始的时间为名称，存储了每一个epoch/iter的训练及验证信息
    • latest.pth即为每一个epoch/iter结束后训练好的参数文件保存
  • result.pkl 由测试进程产生，包含了验证/测试集图片上的测试信息
  • log_analysis 包含对log进行分析产生的结果图片，主要是各种度量关于epoch/iter的变化
  • result_analysis 包含测试结果。其中得分最差的20个图片放在bad文件夹下，得分最好的20个图片放在good文件夹下，这里的所有图片都标注了预测标签和真实标签。
  • demo文件夹下存放用户使用部分数据集之外的图片测试之后产生的结果
