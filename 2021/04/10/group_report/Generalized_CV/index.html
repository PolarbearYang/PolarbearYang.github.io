人工智能普适化平台（CV+深度模型）详细设计文档v0.0

1. 引言

目前已有的数据集、模型等都已经部署于超算平台183.129.217.214:20156。

暂时只能支持PyTorch框架下编写的深度模型。

1.1 编写目的

本文档是人工智能普适化平台（CV+深度模型）的设计文档，
其目的是明确平台的需求和支持项目组内部的研发工作。
该文档描述了人工智能普适化——CV深度模型管理平台DEMO的功能设计，
平台的DEMO将按照这个设计文档进行开发，
适用于本平台的设计人员和开发人员。

为了简便起见，目前仅描述后端运行任务时实现的一些工作。与前端相关的功能暂时省略，有关前端的功能仅提供一些构想。

本设计文档应涵盖人工智能普适化（CV+深度模型）管理平台DEMO的所有功能设计、系统架构设计、接口设计、数据库设计等具体内容。

1.2 功能资料

交互稿待补充。目前暂时用博客代替，预计主要包含以下六大模块(预计仍将继续补充一些模块，打勾的表明该框架下所有任务类型跑通，未打勾则表明该框架下部分任务仍未跑通)：

  • 图像分类
  • 目标检测
  • 语义分割
  • 图像编辑
  • 目标跟踪
  • 动作检测

以上六个模块分别对应留个大类任务，每个任务下还有对应的子任务，如下：

  • mmclassification（图像分类）
    • 单标签分类
    • 多标签分类
  • mmdetection（目标检测）
    • Object Detection
    • Instance Segmentation
  • mmsegmentation（语义分割）
    • Semantic Segmentation（语义分割）
  • mmediting（图像编辑）
    • Super-Resolution（超分辨率）
    • Image Generation（图像生成，目前只有风格迁移）
    • Image Inpainting（图像修复）
    • Image Matting（抠图）
  • mmtracking（目标追踪）
    • Video Object Detection（视频目标检测）
    • Single Object Tracking（单目标追踪）
    • Multi Object Tracking（多目标追踪）
  • mmaction2（动作检测）
    • Action Recognition（动作识别）
    • Action Localization（动作定位）

1.3 软硬件环境

由于项目处于起步阶段，软硬件条件均较为有限。目前使用的硬件仅有一张Tesla V100。目前使用CUDA 11.0、PyTorch 1.7与mmcv 1.2.7。

1.4 准备工作（仅第一次配置环境时使用）

安装PyTorch

pip install torch==1.7.0 torchvision==0.8.0 torchaudio==0.7.0

建立mmcv基本库

pip install mmcv-full==1.2.7 -f https://download.openmmlab.com/mmcv/dist/cu102/torch1.7.0/index.html

建立mmcls本地库

cd mmlab/mmclassification
pip install -e .

建立mmdet本地库

cd mmlab/mmdetection
pip install -r requirements/build.txt
pip install -v -e .

建立mmsegmentation本地库

cd /mmlab/mmsegmentation
pip install -e .

建立mmedit本地库

cd mmlab/mmediting
pip install -v -e .

建立mmaction2本地库

cd mmlab/mmaction2
pip install -r requirements/build.txt
pip install -v -e .

建立mmtrack本地库

cd mmlab/mmtracking
pip install -r requirements.txt
pip install -v -e .

3. 系统架构设计

3.1 设计目标、原则和约束

本项目系统架构设计主要目标在于完成项目业务的需求，实现CV+深度模型普适化平台的基本功能。
这些基本功能包括：数据集的上传与管理、模型的推荐与管理、任务配置制定和运行。

3.2 逻辑架构设计

3.2.1 设计思想

由于在不同任务中，模型的架构和运行流水线、使用的tricks、能够产生的输出，甚至要求数据集提供的信息相差均较大，
我们使用mmlab的方式，先按照运行流程的相似性将任务分为若干个大类，然后再具体细分至每一个任务。这样各个大类任务之间的调整是相互独立的，不会影响相关性较低的任务，便于拓展和维护。

4. 调用配置

比起接口，这一节主要介绍与我们运行完整的任务流程相关的内容。

为了便于前端想向后端传递信息，配置文件都写成了字典的形式。

有些不便展开的细节我们将按照任务大类分文档维护。

对于所有模块，type即调用的python类名，同一模块下的其他参数均是该类的初始化参数。

4.1 数据集

暂未将其制成数据库。我们目前用一个json文件存储数据集信息。

这些信息主要包括程序运行必要的信息，如数据集ID(id)、解析类型(type)、数据集类别(categories)、可用的任务类型(task)、数据集路径(path)、额外参数(CLASSES)。

这些信息中：

  • id是数据集的唯一标识符
  • dataset_name是数据集的名称
  • categories表示数据集可用于哪个领域。目前这个值没用，只是为拓展到自然语言处理等其他任务做准备。
  • task表示该数据集可用的任务类型。这个值可用于决定使用哪个框架下的训练流程。

  • path表示数据集的存储路径
  • additional_args表示一些可能额外输入的参数。比如自定义的数据集是需要输入类名的。
  • tags仅仅是用于展示给用户看的一些标签

4.2 模型

模型根据不同的任务，有不同的组装方式。若要对模型进行配置，需要根据不同任务类型，将模型的各个部件以dict的形式组装起来。

这些内容将分文档描述。由于各人物间模型配置文件差异较大，难以寻找统一格式，这里不再展开细节，详情请见功能资料中的文档说明。

4.3 任务

在数据集和模型都准备完成后，可以开始进行最终任务配置的组装了。不过我们还需要指定一些超参，以及一些其他的配置才能获得一个完整的配置。在配置文件完成之后，我们依次进行训练、测试、log分析、结果分析四个步骤，来获得我们完成一个任务所需要记录的所有信息。接下来我们将逐步介绍每一步应当如何完成。

由于模型结构差异较大，不同类型的任务调用的接口是不同的。

  • 图像分类(mmcls)
    • 训练mmlab/mmclassification/tools/dist_train.sh
    • 测试mmlab/mmclassification/tools/dist_test.sh
    • log分析mmlab/mmclassification/tools/analyze_logs.py(有BUG)
    • 结果分析mmlab/mmclassification/tools/analyze_results.py(有BUG)
  • 目标检测(mmdet)
    • 训练mmlab/mmdetection/tools/dist_train.sh
    • 测试mmlab/mmdetection/tools/dist_test.sh
    • log分析mmlab/mmdetection/tools/analysis_tools/analyze_logs.py
    • 结果分析mmlab/mmdetection/tools/analysis_tools/analyze_results.py
  • 语义分割(mmseg)
    • 训练mmlab/mmsegmentation/tools/dist_train.sh
    • 测试mmlab/mmsegmentation/tools/dist_test.sh
    • log分析mmlab/mmsegmentation/tools/analysis_tools/analyze_logs.py(有BUG)
    • 结果分析(未制定，可暂用demo推断代替)
  • 图像编辑(mmedit)
    • 训练mmlab/mmediting/tools/dist_train.sh
    • 测试mmlab/mmediting/tools/dist_test.sh
    • log分析(未制定)
    • 结果分析(其功能已经融合在“测试”功能中)
  • 其他任务接口还未调通，暂不列出

具体参数根据不同的人物将略有差异，分文档列出。以下需要解释的是配置文件如何组装，以及任务运行后将得到的文件架构。

4.3.1 任务所应当包含的信息

虽然目前尚未制定，直接使用命令行调用程序。考虑到为了方便前端和后端的通讯，把一些可能必要的信息列为json文件格式。

各个参数解释如下：

  • id是对应于每个任务的唯一标识, 任务所包含的所有信息都在task/{ID}这个文件夹下
  • type用于表明应该用哪个任务大类的训练/测试/log分析/结果分析程序来执行这个配置
  • dataset_id和model_name没有实际作用，主要是为了辨认该任务对应于何种数据集和模型

4.3.3 任务的文件架构

关于任务方面，初始创建任务时，我们现在先在task文件夹下创建一个以任务ID为名称的文件夹。然后进行配置文件config.py的组装。

只要文件夹下有一个完整的config.py文件，就可以开始执行“训练-测试-log分析-结果分析”流程。

当一个完整的“训练-测试-log分析-结果分析”流程结束后，预计文件架构如下：

其中各个文件夹内容为：

  • config.py 包含了调用的数据集、模型等信息的配置文件。是初始任务创建的必要文件，该文件制定完整后则可以进行训练。
  • training 包含了训练产生的临时文件、log文件、以及训练好的模型参数的文件夹。该文件夹在训练开始后产生。
    • log文件以训练开始的时间为名称，存储了每一个epoch/iter的训练及验证信息
    • latest.pth即为每一个epoch/iter结束后训练好的参数文件保存
  • result.pkl 由测试进程产生，包含了验证/测试集图片上的测试信息，应当包括每张图片的得分/类别，以及整个模型的性能信息等。
  • log_analysis 包含对log进行分析产生的结果存放，目前主要是各种度量关于epoch/iter的变化曲线。
  • result_analysis 包含测试结果。其中得分最差的20个图片放在bad文件夹下，得分最好的20个图片放在good文件夹下。对于目标检测/实例分割任务，这里的所有图片都标注了预测框（橙色）和ground truth（蓝色）。对于不同任务可能有所差异（比如对于图像编辑将直接存放测试结果）
