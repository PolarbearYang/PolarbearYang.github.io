语义分割模块(mmseg)

1. 引言

该模块包含的内容主要有：

  • 各种语义分割方法和预训练模型
  • 统一的推断流水线
  • 可以对单图推断产生demo并可视化

每一个任务的核心都是配置文件config.py，所有流程都将依赖这个文件执行。

因此本文档主要介绍的是配置文件的内容，从数据集，到模型，最后是如何使用此配置文件（超参和其他配置部分与设计文档中超参相关配置和其他相关配置相同，因此不再描述）。

该任务大类下只有语义分割一种任务。

如果对如何制定配置文件不感兴趣，只关注如何使用默认配置跑通一个流程，请直接跳至第五节-命令行调用部分。

2. 数据集配置的制定

目前主要使用并且已部署的数据集包括：

  • Cityscapes
  • ADE20K
  • Pascal Context(Pascal VOC 2010)
  • Pascal VOC 2012 + Aug
  • CHASE_DB1
  • HRF(高分辨率指纹)
  • STARE(血管)

每个数据集都有自己的解析类，但这只对配置文件中type属性和标注文件ann_file的选择有影响，大体不影响流水线制定。

2.1 数据集配置

以下是一个Cityscapes数据集进行语义分割的配置例子：

  • samples_per_gpu 每个gpu上分配多少个样本(图片), 该数乘以我们实际使用的gpu数就是batch size
  • workers_per_gpu 每个gpu上分配多少个worker
  • train/val/test 分别对应训练流水线/验证流水线/测试流水线，它还有以下几个重要参数：
    • type 对应用于解析数据集的python类的名称
    • data_root 对应type指明的类，所有数据所在的文件夹
    • img_dir 对应于type指明的类，在相对于data_root指明的路径下，原图像文件的路径
    • ann_dir 对应于type指明的类，在相对于data_root指明的路径下，标注文件的路径(标注文件可以是json或者图像)
    • pipeline 流水线的制定，除去读取和输出数据的部分，主要是一些数据预处理手段

前面的例子为了方便查看及使用，将pipeline信息写在data外面，以预定义的形式展现，这将在之后展开说明。pipeline用一个list来展示。

此外，evaluation模块实际上独立于data模块。但由于和数据集的选择高度相关，从而也写在数据集配置文件中。这里的evaluation只包含interval和metric两个参数：

  • interval表示多少个iter进行一次验证（语义分割任务中一般使用IterBasedRunner）
  • metric表示使用的度量类型（一般使用metric='mIoU'即可）

2.2 流水线配置

流水线用一个list，表示的是对数据依次进行的预处理操作。其中有两个组件是测试/验证/测试阶段均必要的：

  • LoadImageFromFile表示从data_prefix对应的文件夹下载入图片的过程。必须放在流水线的第一步。
  • Collect表示对结果格式的统一。这一步进行后，数据便可以以正确的格式进入模型。必须放在最后一步。

其中训练阶段和验证/测试阶段略有差异：

  • 训练阶段需要添加LoadAnnotations表示对标注文件的读取。一般放在流水线的第二步。
  • 在最后一步collect之前，训练阶段应当使用DefaultFormatBundle两步将图像、语义分割信息均转化为PyTorch的Tesnor格式，验证/测试阶段只需使用ImageToTensor转化图片即可。
  • 验证/测试阶段不应当对图像进行任何随机变换之类的数据增强操作，其他应与训练阶段完全一致。

3. 模型配置的制定

如果要使用自己的数据集，且对模型内部其他细节一概不关心的话，在这一部分要修改的只有两项：decode_head以及auxiliary_head下的num_classes！

语义分割模型的结构大致都相似，其组件内容为：

  • backbone
  • decode_head
  • auxiliary_head(optional)

唯一的例外是PointRend中在backbone的基础上额外加了一个neck(FPN)。

目标检测模块(mmdet)

1. 引言

该模块的功能要比其他模块丰富得多，包含的内容主要有：

  • 用于目标检测任务和实例分割任务的模型
  • 模型的模块化设计
  • 多种高效的SOTA模型
  • 可以分析log并产生度量关于epoch改变的图像
  • 可以分析结果并进行可视化
  • 可以对单图或者视频推断产生demo并可视化

每一个任务的核心都是配置文件config.py，所有流程都将依赖这个文件执行。

因此本文档主要介绍的是配置文件的内容，从数据集，到模型，最后是如何使用此配置文件（超参和其他配置部分与设计文档中超参相关配置和其他相关配置相同，因此不再描述）。

这一步模型配置比较复杂，我们主要将模型分为一阶(one-stage)检测模型和二阶(two-stage)检测模型（暂不考虑多阶段）。

大体上讲，实例分割任务仅仅是在目标检测模型上多加一些元素。

如果对如何制定配置文件不感兴趣，只关注如何使用默认配置跑通一个流程，请直接跳至第五节-命令行调用部分。

2. 数据集配置的制定

目前主要使用并且已部署的数据集包括：

  • COCO2017
  • lvis 1.0（基于COCO2017）
  • PASCAL VOC 2007
  • PASCAL VOC 2012
  • Wider Face

每个数据集都有自己的解析类，但这只对配置文件中type属性和标注文件ann_file的选择有影响，大体不影响流水线制定。以下主要以COCO格式的数据集配置作为例子解释。

2.1 目标检测数据集配置

这一部分主要包含目标检测和实例分割公用的一些配置。以下是一个COCO的数据集进行目标检测的配置例子：

和数据集预处理有关的模块主要集中在data模块中，其中至少应包含以下五项：

  • samples_per_gpu 每个gpu上分配多少个样本(图片), 该数乘以我们实际使用的gpu数就是batch size
  • workers_per_gpu 每个gpu上分配多少个worker
  • train/val/test 分别对应训练流水线/验证流水线/测试流水线，它还有以下几个重要参数：
    • type 对应用于解析数据集的python类的名称
    • ann_file 对应于type指明的类，选择的标注文件的路径
    • img_prefix 对应type指明的类，图像数据所在的文件夹

前面的例子为了方便查看及使用，将pipeline信息写在data外面，以预定义的形式展现，这将在之后展开说明。pipeline用一个list来展示。

此外，evaluation模块实际上独立于data模块。但由于和数据集的选择高度相关，从而也写在数据集配置文件中。这里的evaluation只包含interval和metric两个参数：

  • interval表示多少个epoch进行一次验证（一般只使用EpochBasedRunner）
  • metric表示使用的度量类型（对于实例分割，metric=['bbox', 'segm']）

2.2 实例分割数据集配置

实例分割数据集的配置和目标检测几乎差异，由于大部分功能相同，以下只强调一下实例分割中专有的模块。以下是一个实例分割数据集配置的例子：

2.3 流水线配置

流水线用一个list，表示的是对数据依次进行的预处理操作。其中有两个组件是测试/验证/测试阶段均必要的：

  • Collect表示对结果格式的统一。这一步进行后，数据便可以以正确的格式进入模型。必须放在最后一步。

其中训练阶段和验证/测试阶段略有差异：

  • 训练阶段需要添加LoadAnnotations表示对标注文件的读取。一般放在流水线的第二步。

  • 验证/测试阶段不应当对图像进行任何随机变换之类的数据增强操作，其他应与训练阶段完全一致。

3. 模型配置的制定（这一部分可以暂且略过，不推荐用户修改）

如果要使用自己的数据集，且对模型内部其他细节一概不关心的话，在这一部分要修改的只有两项：bbox_head以及mask_head下的num_classes！

目标检测与实例分割模型的结构大致相似，可能有一些参数或部件上的差异，之后会举例说明。综合起来，这些模型可大致分为二阶段(two-stage)模型和一阶段模型(one-stage)。实际上还存在多阶段模型，这里目前不做讨论。这可能也是制定数据库最困难的一部分。虽然之后想允许前端提供改造模型的接口，但以下内容可能还不够完善，请谨慎阅读！

3.1 二阶段(two-stage)模型

对于二阶段(two-stage)模型，组件内容为：

  • backbone
  • neck
  • rpn head(dense head)
  • roi head(sparse head)
    • bbox roi extractor
    • bbox head
    • mask roi extractor(实例分割专用)
    • mask head(实例分割专用)

在模型配置方面，实例分割和目标检测有一些细节上的差异。接下来先介绍目标检测模型，然后再介绍实例分割模型中与目标检测不同的内容。

3.1.1 目标检测模型

但是，这里还有必要介绍一下各个组件的作用。

3.1.1.1 Backbone与Neck

Backbone承担着特征提取器的作用，通常载入一些（在ImageNet上训练好的）预训练参数。pretrained选项正是为载入参数而准备的。

而Neck通常是FPN、PAFPN等特征金字塔，需要与Backbone逐层输出的大小严格对应（这也是用户很容易犯错的地方），同样也是由type指定类型，其他参数都是对应于该模块类的参数。

3.1.1.2 Region Proposal Network(rpn_head)

这里二阶段(two-stage)模型主要进行的是二步交替训练。该网络的作用是初步产生一些区域，并在train_cfg/test_cfg中nms的辅助下筛选区域，将这些区域输给后面的roi_head。

3.1.1.3 Region Of Interest Head(roi_head)

这一步基于rpn_head输出的区域集，产生最后的结果。这同样需要train_cfg/test_cfg中nms的辅助。

3.1.2 实例分割模型

实例分割相比于目标检测，除了增加了mask_roi_extractor、mask_head，以及对应的train_cfg/test_cfg有所改动之外，其余部分大致相同。

对于一阶段(one-stage)模型，组件内容为：

  • backbone
  • neck
  • bbox head
  • mask_head
  • segm_head

3.3 特别提醒

如果要使用自己的数据集，在这一部分要修改的只有两项：bbox_head以及mask_head下的num_classes！

图像分类模块(mmcls)

1. 引言

该模块包含的内容主要有：

  • 单标签分类和多标签分类
  • 各种backbone和预训练模型
  • tricks的集合
  • 可以对单图推断产生demo并可视化

每一个任务的核心都是配置文件config.py，所有流程都将依赖这个文件执行。

因此本文档主要介绍的是配置文件的内容，从数据集，到模型，最后是如何使用此配置文件（超参和其他配置部分与设计文档中超参相关配置和其他相关配置相同，因此不再描述）。

单标签分类和多标签分类使用的模型架构大致相同，其唯一的区别在于其使用的cls_head不同（单标签使用LinearClsHead，多标签使用MultiLabelClsHead）。

如果对如何制定配置文件不感兴趣，只关注如何使用默认配置跑通一个流程，请直接跳至第五节-命令行调用部分。

2. 数据集配置的制定

目前主要使用并且已部署的数据集包括：

  • ImageNet(ILSVRC2012)
  • CIFAR10
  • CIFAR100
  • MNIST
  • FashionMNIST
  • Pascal VOC 2007(Multi-Label)

每个数据集都有自己的解析类，但这只对配置文件中type属性和标注文件ann_file的选择有影响，大体不影响流水线制定。

2.1 数据集配置

单标签数据集和多标签数据集的差别不大。以下是一个ImageNet数据集进行单标签分类的配置例子：

和数据集预处理有关的模块主要集中在data模块中，其中至少应包含以下五项：

  • samples_per_gpu 每个gpu上分配多少个样本(图片), 该数乘以我们实际使用的gpu数就是batch size
  • workers_per_gpu 每个gpu上分配多少个worker
  • train/val/test 分别对应训练流水线/验证流水线/测试流水线，它还有以下几个重要参数：
    • type 对应用于解析数据集的python类的名称
    • data_prefix 对应type指明的类，图像数据所在的文件夹
    • ann_file 对应于type指明的类，选择的标注文件的路径
    • pipeline 流水线的制定，除去读取和输出数据的部分，主要是一些数据预处理手段

此外，evaluation模块实际上独立于data模块。但由于和数据集的选择高度相关，从而也写在数据集配置文件中。这里的evaluation只包含interval和metric两个参数：

  • interval表示多少个epoch进行一次验证（一般只使用EpochBasedRunner）
  • metric表示使用的度量类型（对于单标签分类，metric='accuracy'；对于多标签分类，metric=['mAP', 'CP', 'OP', 'CR', 'OR', 'CF1', 'OF1']）

多标签分类的配置格式除了evaluation项外与单标签分类完全相同，这里不再列出。

2.2 流水线配置

流水线用一个list，表示的是对数据依次进行的预处理操作。其中有两个组件是测试/验证/测试阶段均必要的：

  • LoadImageFromFile表示从data_prefix对应的文件夹下载入图片的过程。必须放在流水线的第一步。
  • Collect表示对结果格式的统一。这一步进行后，数据便可以以正确的格式进入模型。必须放在最后一步。

其中训练阶段和验证/测试阶段略有差异：

  • 在最后一步collect之前，训练阶段应当使用ImageToTensor和ToTesnor两步将图像和标签均转化为PyTorch的Tesnor格式，验证/测试阶段只需转化图片即可
  • 验证/测试阶段不应当对图像进行任何随机变换之类的数据增强操作，其他应与训练阶段完全一致。

Node.js安装

首先就是node.js的安装失败，遇到了一个vs2017无法安装的问题。

本人一开始直接把下载的node.js的Windows Binary版本解压到本地，运行了install_tools.bat文件，然后就是各种vs2017相关的安装失败，查询博客无果，开管理员权限也无效。一怒之下，直接选择添加至环境变量解决了该问题。本人先将NODE_PATH加入系统环境变量，其值为：

D:\Program Files\node-v14.16.0-win-x64

并在Path中加入%NODE_PATH%。再在cmd中测试node -v和npm -v，发现问题竟然解决了。之前这两个指令一直显示“command not found”。

后来按照某篇博客，加入node_global和node_cache两个文件夹，并将%NODE_PATH%\node_global加入系统变量Path。并打开cmd执行以下两条指令：

npm config set prefix "D:\Program Files\node-v14.16.0-win-x64\node_global"   
npm config set cache "D:\Program Files\node-v14.16.0-win-x64\node_cache"

Hexo的安装

本人一开始使用了如下安装命令：

npm install hexo-cli -g

发现hexo -v命令无效，找不到Hexo。后来只好手动寻找了hexo的位置，后来竟然在D:\Program\下找到了hexo，并手动将该路径加入系统环境变量Path解决了问题。记得还需要安装chalk：

npm install chalk

以及

npm install hexo-deployer-git --save
